# Token-Based Auth: Architecture & FAQ

---

## Architecture Overview

### Three Auth Modes in Single Codebase

```
┌─────────────────────────────────────────────────────────────┐
│  MASTER (Admin)                                             │
│  ├─ Env Vars: VIEWER_USERNAME + VIEWER_PASSWORD             │
│  ├─ Token Hash: PBKDF2(username:password)                   │
│  ├─ Session: Stored in _viewer_sessions dict                │
│  └─ Access: All chats (respects DISPLAY_CHAT_IDS if set)    │
├─────────────────────────────────────────────────────────────┤
│  VIEWER (DB-Backed Account)                                 │
│  ├─ Username + Password stored in viewer_accounts table     │
│  ├─ Password: PBKDF2-SHA256 hashed                          │
│  ├─ Allowed chats: Per-account whitelist (JSON)             │
│  ├─ Session: Token → _viewer_sessions dict + DB persistent  │
│  └─ Created by: Admin via setup script or API               │
├─────────────────────────────────────────────────────────────┤
│  SHARE TOKEN (NEW)                                          │
│  ├─ Generated by: Admin via POST /api/admin/tokens          │
│  ├─ Token: secrets.token_hex(32) shown once at creation     │
│  ├─ Storage: PBKDF2-SHA256 hash + salt in viewer_tokens     │
│  ├─ Allowed chats: Per-token whitelist (JSON)               │
│  ├─ Expiry: Optional (NULL = no expiry)                     │
│  ├─ Revocation: Instant (is_revoked flag)                   │
│  ├─ Session: Token → session token → _viewer_sessions dict  │
│  └─ Use case: Share links, demos, audits, temporary access  │
└─────────────────────────────────────────────────────────────┘
```

### Authentication Flow Diagram

```
User Request
    │
    ├─ Has Cookie? ─yes─> Check _viewer_sessions dict
    │                      │
    │                      ├─ Valid & not expired? ─yes─> Grant access
    │                      └─ Invalid/expired? ─────────> Return 401
    │
    ├─ Has Token (query/header)? ─yes─> DB: verify_viewer_token()
    │                                     │
    │                                     ├─ Hash matches?
    │                                     ├─ Not revoked?
    │                                     ├─ Not expired?
    │                                     │
    │                                     ├─ All yes? ──> Create session + issue cookie
    │                                     └─ Any no? ───> Return 401
    │
    └─ No cookie/token? ────────> Return 401 (if AUTH_ENABLED)
```

## Comparison: All Three Auth Modes

| Aspect | Master (Env Vars) | Viewer (DB Account) | Share Token (NEW) |
|--------|---|---|---|
| **Creation Method** | Environment variables | Admin API or setup script | Admin: POST /api/admin/tokens |
| **Storage** | None (derived from env vars) | viewer_accounts table | viewer_tokens table |
| **Identity** | VIEWER_USERNAME | viewer_accounts.username | token_id (anonymous) |
| **Lifespan** | Permanent | Permanent until disabled | Time-bound (optional) |
| **Allowed Chats** | DISPLAY_CHAT_IDS or all | Per-account whitelist (JSON) | Per-token whitelist (JSON) |
| **Multiple Users** | Single admin | Multiple viewers | Multiple viewers (anonymous) |
| **Revocation** | Disable env vars (restart needed) | Disable account | Revoke token (instant) |
| **Audit Trail** | Logged as master | Logged with viewer_id | Logged as token_id |
| **Password** | Required | Required (hashed) | Not needed |
| **Use Case** | Primary admin access | Permanent viewer accounts | Temporary/demo access |
| **Delegation** | Not delegated | Admin creates accounts | Admin creates share links |
| **Session Persistence** | In-memory + DB | In-memory + DB | In-memory + DB |

## Security Comparison: Token Storage Methods

### Option A: Hash Token (Recommended for This Project)

```
Plaintext token: "abc123xyz..."  (shown to user once)
                    ↓
            PBKDF2-SHA256 hash
                    ↓
Database storage: "a1f2b3c4d5..." (only hash, no plaintext)
```

**Pros:**
- Database breach doesn't expose plaintext tokens
- One-way function; can't reverse hash to get original token
- Matches existing codebase (password hashing uses same pattern)
- OWASP compliant

**Cons:**
- Can't retrieve lost tokens (no "show me my token" feature)
- Requires salt storage for verification

**Security level:** ⭐⭐⭐⭐⭐

---

### Option B: Plaintext Token (Not Recommended)

```
Plaintext token: "abc123xyz..."
                    ↓
Database storage: "abc123xyz..." (raw string)
```

**Pros:**
- Simpler (no hashing)
- Can retrieve lost tokens

**Cons:**
- Database breach exposes ALL tokens immediately
- Attacker can impersonate any viewer
- Violates OWASP guidance
- Does not match codebase patterns

**Security level:** ⭐

---

### Option C: Encrypt Token (Alternative, Not Recommended for MVP)

```
Plaintext token: "abc123xyz..."
                    ↓
        AES-256 encryption (with key)
                    ↓
Database storage: "xyzabc..." (encrypted ciphertext)
```

**Pros:**
- Retrievable (decrypt with key)
- More secure than plaintext

**Cons:**
- Requires key management (where to store key?)
- More complex than hashing
- If key stored on same server, attacker gets both key and ciphertext
- Overkill for token use case

**Not recommended:** Hashing is simpler and sufficient.

## URL Parameter Security: Detailed Risk Analysis

### Scenario 1: User Shares Link Publicly

```
User sends: "https://domain/?token=abc123xyz" to friend via Slack
                                      ↑
                    Token visible in message (shared)
```

**Mitigation:** Not much we can do here (user choice). Best practice: educate users to share via secure channel and use expiry dates.

---

### Scenario 2: Browser History Attack

```
User visits: https://domain/?token=abc123xyz
                             ↑
            Stored in browser history (searchable)

    Attacker gains access to user's browser
        ↓
    Searches history for "token="
        ↓
    Finds all tokens user has ever accessed
        ↓
    Can use tokens to impersonate user
```

**Mitigation:** Redirect to same URL without token after login.

```python
# User visits: https://domain/?token=abc123xyz
# Server validates token
# Server issues session cookie
# Server redirects to: https://domain/  (no token)
# Browser history shows: https://domain/  (safe)
```

**Effectiveness:** ✅ High. Token never stored in history.

---

### Scenario 3: Server Log Leakage

```
User visits: https://domain/?token=abc123xyz
                             ↑
            Server logs HTTP request:
            "GET /?token=abc123xyz HTTP/1.1"
            ↓
    Attacker gains read access to server logs
        ↓
    Finds token in plaintext
        ↓
    Can use token to access viewer
```

**Mitigation:** Avoid query parameters entirely for sensitive data. Use POST body or Bearer header.

```python
# Option 1: POST form (token in body, not URL)
POST /auth/token
Content-Type: application/x-www-form-urlencoded

token=abc123xyz
    ↓
Not logged in URL logs (body not typically logged)

# Option 2: Bearer header
Authorization: Bearer abc123xyz
    ↓
May still be logged, but considered credentials (some middleware redacts)

# Option 3: Query param with immediate redirect
GET /?token=abc123xyz
    ↓
Issue cookie
    ↓
Redirect to /  (new GET logged as "GET / HTTP/1.1")
    ↓
Mitigation: Token visible only once in logs, then not reused
```

**Recommendation for this project:** Use Option 3 (query param + redirect). Trade-off:
- Token visible in logs once (at initial login)
- Never reused (only valid once, then session cookie used)
- User never needs to paste token again
- Simple UX (click share link = logged in)

---

### Scenario 4: Referer Header Leakage

```
User has browser at: https://domain/?token=abc123xyz
User clicks link to: https://external-site.com
                        ↑
            Browser sends Referer header:
            "Referer: https://domain/?token=abc123xyz"
                        ↓
    external-site.com sees the token!
        ↓
    Can use token to access viewer (if token not time-bound)
```

**Mitigation:** Set `Referrer-Policy: no-referrer` header.

```python
response.headers["Referrer-Policy"] = "no-referrer"
```

**Effectiveness:** ✅ High (blocks Referer header). But only works if:
1. Page served with header (must set in FastAPI endpoint)
2. External domain receives request (can't control their side)

**Better mitigation:** Redirect away from token URL immediately (Scenario 2 solution).

## FAQ: Implementation Questions

### Q1: Should tokens support multiple concurrent sessions?

**Current design:** No limit. Single token can have multiple sessions (different browsers).

**Pros:** Flexibility, user can open token on phone + desktop
**Cons:** Can't revoke all sessions at once by revoking token

**Recommendation:** Current design is fine for MVP. Add `max_sessions` column if needed later.

---

### Q2: Can users rotate/refresh tokens?

**Current design:** No. Admin must revoke and create new.

**Options:**
1. Admin revokes old, creates new (current)
2. Auto-generate new token on first use (complex)
3. Refresh endpoint: POST /api/tokens/{id}/refresh (generates new, invalidates old)

**Recommendation:** Option 1 (admin-driven). Simple, auditable, secure.

---

### Q3: What if user loses the token before using it?

**Current design:** No retrieval. Token shown once; if user loses it, they lose access.

**Why:** Hashing is one-way; we can't decrypt to show plaintext later.

**Workaround:** Admin can revoke old token and create new one.

**Recommendation:** Educate users: "Copy token immediately; save in password manager or send securely."

---

### Q4: Should tokens support granular permissions (e.g., read-only)?

**Current design:** Binary. Token grants full access to allowed chats (no read-only mode).

**Future enhancement:** Add `permissions` column (JSON): `{"chats": ["123"], "actions": ["read"]}`

**Recommendation:** Out of scope for MVP. Add for v2.

---

### Q5: How do we prevent token brute-force attacks?

**Current design:** No rate limiting.

**Mitigation options:**
1. **Simple:** Rate limit by IP on POST /auth/token endpoint
2. **Better:** Log failed attempts, lock after N failures
3. **Best:** WAF (Web Application Firewall) rules

**Recommendation:** Implement rate limiting on verify endpoint (e.g., 5 attempts per IP per minute).

```python
@app.get("/auth/token")
async def auth_with_token(request: Request, token: str):
    # Check rate limit by IP
    ip = request.client.host
    if rate_limiter.is_limited(ip):
        raise HTTPException(status_code=429, detail="Too many attempts")

    # ... rest of code
```

---

### Q6: Should we hash the token or use JWT?

**Our choice: Hash (not JWT)**

| Aspect | Hashed Token | JWT |
|---|---|---|
| **Storage** | Hash in DB | No DB needed (stateless) |
| **Revocation** | Instant (flag-based) | Wait for expiry (stateless) |
| **Database** | Requires DB lookup | No DB lookup needed |
| **Scalability** | DB queries per request | No DB queries |
| **Complexity** | Simple | Medium |

**Why hash for this project:**
- Instant revocation needed (admin can disable token immediately)
- Database already exists (viewer_tokens table)
- Codebase pattern already established (passwords hashed same way)

**JWT would be better if:**
- Multi-server deployment (stateless)
- No database available
- Revocation not important

---

### Q7: Can admins see token plaintext after creation?

**Answer:** No. Design choice.

**Rationale:**
- Security best practice (tokens shown once only)
- Prevents accidental leakage (if DB query results shown in logs)
- Encourages users to save token immediately

**Workaround:** Revoke + create new token (new token shown).

---

### Q8: How do we log token usage for audit trail?

**Implementation:**

```python
# When creating session from token:
await db.log_token_usage(
    token_id=viewer_info["id"],
    ip_address=request.client.host,
    user_agent=request.headers.get("user-agent"),
    action="login"
)

# When accessing chats with token session:
# (Existing audit logging already logs via request.state.user["viewer_id"])
```

**Audit trail:**
- `viewer_tokens` table tracks `created_at`, `last_used_at`, `created_by`
- `viewer_audit_log` table tracks each API request (existing)

**Result:** Admin can see which token was created when, who created it, when last used.

---

### Q9: Should we rotate session cookies for token-based auth?

**Current design:** No rotation. Session issued once, used until expiry or logout.

**Why:** Session token is separate from share token (not conflated).

**Recommendation:** No change needed. Existing session TTL (24 hours) provides security.

---

### Q10: Can we use short tokens (less than 32 bytes)?

**Answer:** Not recommended.

**Math:**
- 32 bytes = 256 bits = 2^256 possible values
- 16 bytes = 128 bits = 2^128 possible values (still very large)
- Collision probability negligible at both sizes

**Why 32 bytes:**
- OWASP standard (2015+)
- Consistent with codebase (passwords use 32-byte salts)
- Negligible performance difference
- Tokens are shared via URL (short vs. long doesn't matter)

**Recommendation:** Stick with 32 bytes.

## Architecture Summary

```
Existing Auth (Password)          New Auth (Share Token)
─────────────────────────────     ──────────────────────
Admin creates viewer account  →   Admin generates share link
User: username + password     →   User: token (no password)
Stores in viewer_accounts     →   Stores in viewer_tokens
Session token generated       →   Session token generated
Cookie issued                 →   Cookie issued
Access to allowed chats       →   Access to allowed chats

Both converge on:
  session_token → _viewer_sessions dict
              → viewer_sessions DB table
              → Cookie (HttpOnly)
```

## Performance Impact

### Database Queries

**Per token verification:**
1. SELECT from viewer_tokens WHERE is_revoked=0
2. Loop through records, compare hash (Python, no DB query)
3. UPDATE last_used_at (optional)

**Cost:** ~1-2 DB queries per initial token authentication (then cached in session).

**Typical load:**
- 1000 concurrent sessions: minimal (in-memory)
- 100 new token authentications/minute: 100-200 DB queries/minute (negligible)

**Optimization:** Index on `is_revoked` + `created_by` for admin list queries.
